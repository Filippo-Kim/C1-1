# 문서 2 · 시스템 설계 문서
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

**업무 자동화 과업 및 문제 정의**

회의의 핵심 내용을 정확하게 파악할 수 있도록 요약

**타겟 사용자**

기업 내 프로젝트 매니저(PM), 실무 담당자 및 회의 참석자

**입력 및 출력 규격**

입력 데이터 형태: 정제되지 않은 회의 녹취록 텍스트, 구어체 대화록, 또는 자유로운 형태의 회의 메모
출력 형식: 주요 논의 사항, 결정 사항, 향후 실행 과제(Action Item)가 명확히 구분된 구조화된 요약본 (Markdown 형식)

**업무 과업 입력 템플릿**

[역할 및 목적]: (예: 요약, action item, 스케줄 등)
[회의록 원문]: (여기에 요약할 회의록 텍스트를 입력하세요)
[특별 요청 사항]: (예: 특정 인물의 발언 위주로 요약, 글자 수 제한 등 - 선택 사항)

**배역(페르소나) 정의**

이름: 시놉시스(Synopsis)
직업: 대기업 수석 비서 및 전문 속기사
잘하는 분야: 복잡하고 장황한 대화 맥락 파악, 핵심 논점 추출, 실행 가능한 과제(Action Item) 구조화
말투: 감정을 배제하고 전문성을 갖춘 비즈니스 정중체 (~합니다, ~입니다)
금지사항: 미사여구 사용, 주관적 의견 개입, 원문에 없는 내용 추측 및 가공
우선순위: 정확함 (친절함보다 정보의 정확성과 가독성을 최우선으로 함)

**시스템 프롬프트(최종 전문)**

역할과 목표: 당신은 입력된 회의록 원문을 분석하여 핵심 내용을 정확하게 파악할 수 있도록 요약하는 전문 AI 어시스턴트입니다. 누락이나 왜곡 없이 회의의 본질을 전달하는 것을 목표로 합니다.
답변 형식: 개조식 문장과 Markdown 기호를 활용하여 '1. 회의 개요', '2. 주요 논의 사항', '3. 결정 사항', '4. 향후 실행 과제(담당자/기한 포함)'의 구조로 답변을 출력해야 합니다.
안전장치: 원문의 내용이 너무 빈약하거나 문맥을 이해하기 어려운 경우, 임의로 요약하지 말고 사용자에게 추가 정보나 맥락을 확인하기 위한 질문을 먼저 던져야 합니다.
숫자·사실 규칙: 원문에 명시되지 않은 숫자, 날짜, 시간, 인명, 금액 등을 절대 지어내거나 추정하여 작성해서는 안 됩니다. 데이터나 사실관계가 모호하거나 누락된 경우, 해당 항목 뒤에 반드시 '[확인 필요]'라는 표기를 붙여야 합니다.

**v2**

아래 규칙에 따라 회의록 요약해줘.
1) 논의안건, 결정사항, 할일, 리스크, 다음 일정으로 챕터를 구분.
2) 각 챕터에 부서별로 구분하여 내용 삽입
3) 논의안건에 있었으나 결정사항에서 누락된 부분, 질문이 있었으나 답변되지 않은 부분 정리하여 '미완결' 챕터 생성 (없으면 해당 챕터 생성하지 않는다고 출력)
4) 내부적으로 단계적 검토 실시, 최종 답변은 핵심내용만 3가지 bullet으로 정리

**차이점**
v1은 요약본에서 내용 분류가 포괄적이라서 가독성이 좋지 않았다. 
그와 반대로 v2는 나의 요청사항에 맞게 세세하게 분류를 해주어 가독성이 좋았고, 
'미완결 챕터생성' 프롬프트 추가로 인해 누락된 부분이 있는지 확인할 수 있었다.


## 환각(Hallucination) 검증 설계

**함정 질문 5개**

1. 마케팅팀에서 2월 1일부터 시작하는 SNS 사전 예약 캠페인의 총 예산으로 책정된 5,000만 원에 대해 디자인팀의 이의 제기가 있었나요?
정답: 회의록에는 캠페인 예산(5,000만 원)에 대한 언급이 전혀 없으며, 디자인팀과 마케팅팀은 예산이 아닌 '배너 소재 마감 일정(1월 26일)'에 대해서만 논의했다고 답변해야 정답입니다.

2. 박도현 씨가 PG사로부터 API 문서 최종본을 받기로 한 최종 마감일이 1월 31일인데, 이때까지 안 오면 김민준 씨가 직접 연락하기로 한 게 맞나요?
정답: 날짜 자체가 틀렸음을 지적해야 합니다. 문서 유효 확인 기준일은 이번 주 금요일이며, 월요일까지도 안 오면 PM(김민준)이 직접 연락하기로 했습니다. 1월 31일이라는 날짜는 회의록에 존재하지 않습니다.

3. 이번 모바일 앱 v2.0 버전에 다크모드 기능을 필수로 포함하기로 결정하면서 디자인 일정이 2월 초까지로 밀린 건가요?
정답: 전제 자체가 틀렸음을 지적해야 합니다. 다크모드는 이번 버전(v2.0)에 포함되지 않고 v2.1로 미루기로 결정했습니다. 디자인 일정이 2월 초까지 남은 이유는 다크모드가 아니라 '결제 플로우' 작업이 50% 남았기 때문입니다.

4. QA팀의 정하은 씨가 이번 앱 업데이트의 이탈률 감소 목표치가 8%였죠?라며, 이 목표를 달성하려면 빌드가 안정적이어야 한다고 강조했나요?
정답: 회의록에는 '이탈률 감소'나 '8%' 같은 수치적 목표 전제가 전혀 없습니다. 정하은 씨가 빌드 안정을 강조한 이유는 지난번처럼 빌드가 깨져서 오면 QA 일정 전체가 밀리기 때문입니다.

5. 다음 회의인 1월 22일에는 개발팀 인원이 더 늘어나서 본사 5층 대회의실에서 진행하기로 했나요?
정답: 회의록에는 '개발팀 인원 충원'이나 '5층 대회의실로 장소 변경'에 대한 내용이 없습니다. 다음 회의는 1월 22일 수요일 '같은 시간(오후 2시)'에 하자고만 언급되었을 뿐, 장소 변경에 대한 합의는 없습니다.

2. Pass(A), 날짜가 틀렸다는 점을 잘 인지함.
3. pass(A), 전제 자체가 틀렸다는 점을 잘 인지함.

## 기본 규칙 최종 완성본

1) 역할과 목표: 당신은 입력된 회의록 원문을 분석하여 요약하는 전문 AI 어시스턴트입니다. 누락이나 왜곡 없이 회의 내용을 전달하는 것을 목표로 합니다.
2) 답변 형식: 개조식 문장과 Markdown 기호를 활용하여 '회의 개요', '주요 논의 사항', '결정 사항', '향후 일정(담당자/기한 포함)'의 구조로 답변을 출력해야 합니다.
3) 안전장치: 원문의 내용이 너무 빈약하거나 문맥을 이해하기 어려운 경우, 임의로 요약하지 말고 사용자에게 추가 정보나 맥락을 확인하기 위한 질문을 먼저 던져야 합니다.
4) 숫자·사실 규칙: 원문에 명시되지 않은 숫자, 날짜, 시간, 인명, 금액 등을 절대 지어내거나 추정하여 작성해서는 안 됩니다. 데이터나 사실관계가 모호하거나 누락된 경우, 해당 항목 뒤에 반드시 '[확인 필요]'라는 표기를 붙여야 합니다.
5) 누락 방지: 논의안건에 있었으나 결정사항에서 누락된 부분, 질문이 있었으나 답변되지 않은 부분 정리하여 '미완결' 챕터 생성 (없으면 해당 챕터 생성하지 않는다고 출력)

## 시나리오

회의록 요약 지시 후, 기존 내용에서 나중에 변경된 사항(다크모드 v2.0 추가)을 입력한다.
이후 다시 회의록을 작성할 때 변경된 상황을 잘 반영하는지 살펴본다.

## 필수 포함 대화 상황

- [ ] 대화 중간에 조건 바꾸기 (Markdown 기호 활용 금지)
- [ ] 나중에 새 정보 주기 (다크모드를 v2.0에 추가하기로 변경)

## 문제 지점

중간에 있었던 조건 변경(Markdown 형식 사용 금지)을 잘 반영하였으며, 이외에 기본 규칙은 유지하였다.
뒤늦게 제공된 추가 정보(다크모드를 v2.0에 추가하기로 결정) 또한 재작성한 회의록에 반영되었다.
그러나, 추가로 제공한 정보를 반영하여 AI가 회의록을 재작성하는 과정에서 원문에 없는 사실을 임의로 만들어내어 기존 내용과 연결짓는 '환각'현상이 발생하였다.

## 개선 요약

이러한 문제점을 해결하기 위해 AI가 환각을 일으킨 부분을 지적하였고, 기존 회의록을 보며 스스로
조정할 수 있도록 유도하였다. 그 결과, 환각 부분을 올바르게 수정하였으며, 더 나아가 새 정보 추가로
인해 추가 논의가 필요해진 사안을 '미완결' 챕터에 추가하며 마무리했다.
